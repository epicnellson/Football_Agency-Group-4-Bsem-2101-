1. SYSTEM OVERVIEW
================================================================================

1.1 PROJECT DESCRIPTION
----------------------
The Football Agent Management System is a comprehensive web-based application designed
to manage football players, agents, and club managers. The system provides a
centralized platform for user management, profile administration, and contact
handling with role-based access control.

1.2 SYSTEM PURPOSE
------------------
Primary objectives:
- Manage multiple user roles with different permission levels
- Provide secure authentication and authorization
- Enable efficient user and player profile management
- Facilitate communication through contact forms
- Maintain data integrity and security

1.3 TARGET USERS
----------------
- System Administrators: Full system control and user management
- Players: Profile management and statistics tracking
- Agents: Player representation and client management
- Club Managers: Team and player scouting operations

1.4 TECHNICAL STACK
--------------------
Frontend: HTML5, CSS3, JavaScript
Backend: PHP 7.4+, MySQLi
Database: MySQL 5.7+ / MariaDB 10.2+
Server: Apache/Nginx
Security: Password hashing, CSRF protection, input validation

3. DATABASE DESIGN
================================================================================

3.1 DATABASE SCHEMA
-------------------
Database Name: football_agent_db
Character Set: utf8mb4
Collation: utf8mb4_general_ci

3.2 TABLE STRUCTURES
--------------------

TABLE: users
------------
Purpose: Stores user authentication and basic profile information
Columns:
- id (INT, PRIMARY KEY, AUTO_INCREMENT)
- username (VARCHAR(50), UNIQUE)
- email (VARCHAR(100), UNIQUE)
- password (VARCHAR(255), HASHED)
- role (ENUM: 'admin', 'player', 'agent', 'club_manager')
- first_name (VARCHAR(50))
- last_name (VARCHAR(50))
- phone (VARCHAR(20))
- address (TEXT)
- profile_image (VARCHAR(255))
- is_active (TINYINT(1), DEFAULT 1)
- created_at (TIMESTAMP, DEFAULT CURRENT_TIMESTAMP)
- updated_at (TIMESTAMP, DEFAULT CURRENT_TIMESTAMP ON UPDATE)

TABLE: players
--------------
Purpose: Stores player-specific football information
Columns:
- id (INT, PRIMARY KEY, AUTO_INCREMENT)
- user_id (INT, FOREIGN KEY references users.id)
- date_of_birth (DATE)
- position (ENUM: 'Goalkeeper', 'Defender', 'Midfielder', 'Forward')
- height (DECIMAL(5,2))
- weight (DECIMAL(5,2))
- preferred_foot (ENUM: 'Left', 'Right', 'Both')
- current_club (VARCHAR(100))
- agent_id (INT, FOREIGN KEY references users.id)
- video_url (VARCHAR(255))
- stats (TEXT, JSON format)

TABLE: agents
-------------
Purpose: Stores agent-specific information
Columns:
- id (INT, PRIMARY KEY, AUTO_INCREMENT)
- user_id (INT, FOREIGN KEY references users.id)
- license_number (VARCHAR(50))
- years_experience (INT)
- specialization (TEXT)
- total_players (INT, DEFAULT 0)

TABLE: club_managers
-------------------
Purpose: Stores club manager information
Columns:
- id (INT, PRIMARY KEY, AUTO_INCREMENT)
- user_id (INT, FOREIGN KEY references users.id)
- club_name (VARCHAR(100))
- club_location (VARCHAR(100))
- club_level (ENUM: 'Local', 'National', 'International')

3.3 ENTITY RELATIONSHIPS
------------------------
- users (1) -> players (1): One user can have one player profile
- users (1) -> agents (1): One user can have one agent profile
- users (1) -> club_managers (1): One user can have one club manager profile
- agents (1) -> players (N): One agent can represent multiple players
- users.id -> players.user_id (Foreign Key)
- users.id -> agents.user_id (Foreign Key)
- users.id -> club_managers.user_id (Foreign Key)
- users.id -> players.agent_id (Foreign Key)
